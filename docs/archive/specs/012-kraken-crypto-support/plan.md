# Implementation Plan: Kraken Crypto Data Support

**Branch**: `012-kraken-crypto-support` | **Date**: 2026-02-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/012-kraken-crypto-support/spec.md`

## Summary

Add Kraken cryptocurrency exchange as a data source for backtesting. A new `KrakenHistoricalClient` fetches historical OHLCV data via the Kraken Futures Charts API (which supports spot data with date-range pagination), converts it to Nautilus `Bar` and `CurrencyPair` objects, and stores it in the existing Parquet catalog. The `DataCatalogService` is extended with a `data_source` parameter to route fetching through either the IBKR or Kraken client. Configuration follows the existing `IBKRSettings` pattern with env-var-only credential management.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: python-kraken-sdk, nautilus-trader, FastAPI, Pydantic
**Storage**: Parquet (via Nautilus ParquetDataCatalog), PostgreSQL/TimescaleDB (existing backtest results)
**Testing**: pytest (unit → component → integration with `--forked` → e2e)
**Target Platform**: macOS / Linux
**Project Type**: CLI + web-service (extending existing)
**Performance Goals**: <2s cached data retrieval, zero Kraken rate-limit violations
**Constraints**: Kraken Spot OHLC API limited to 720 most-recent entries; use Futures Charts API for historical depth
**Scale/Scope**: Single user, multiple crypto pairs, months/years of historical data

## Key Design Decisions

### D-001: Futures Charts API over Spot OHLC for Historical Data

The Spot OHLC endpoint returns max 720 entries (most recent only). The Futures Charts API (`/api/charts/v1/spot/:symbol/:resolution`) supports date ranges with pagination, making it suitable for backtesting. See [research.md](research.md#r-001) for full analysis.

### D-002: Extend DataCatalogService Instead of Creating New Service

Adding a `data_source` parameter to `fetch_or_load()` and a lazy `kraken_client` property reuses all existing caching, gap detection, and Parquet logic. Creating a separate service would duplicate ~800 lines. See [research.md](research.md#r-006).

### D-003: Build CurrencyPair Instruments from Kraken AssetPairs API

Unlike IBKR (which returns Nautilus Instrument objects), Kraken requires manual construction. The `/0/public/AssetPairs` endpoint provides precision, min order size, and fee data. Instrument definitions are cached after first fetch. See [research.md](research.md#r-004).

### D-004: User-Facing Pair Names Use Standard Symbols (BTC not XBT)

Users specify `BTC/USD`, not Kraken's native `XXBTZUSD` or `XBT/USD`. A mapping layer in the client handles conversions. See [research.md](research.md#r-003).
